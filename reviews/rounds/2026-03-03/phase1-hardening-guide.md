# Phase 1 — Hardening Guide Audit (@HardeningGuideAgent, Sonnet)

Date: 2026-03-03

## Summary

Audit of `WordPress-Security-Hardening-Guide.md` against `AGENTS.md` constraints, `WP-Security-Style-Guide.md` normative rules, and `WordPress-Security-Benchmark.md` cross-document alignment.

**Total findings: 12**

| Severity | Count |
|---|---|
| Critical | 1 |
| High | 4 |
| Medium | 5 |
| Low | 2 |

The most significant issues are: a technically incorrect claim about where WordPress roles can be defined in code (Critical); a cross-document URL mismatch on NIST SP 800-63B revision (High); a probable stale IBM cost figure carried over from a prior year's report (High); a citation from a non-authoritative commercial vendor source (High); and the NIST SP 800-61 reference pointing to a superseded revision (High). Several Medium findings address vague compliance references, an unverifiable OWASP edition, and an internal citation inconsistency. No deprecated constants are used without annotation. No markdown escaping errors were found inside inline code spans or prose.

## Findings

### Critical

| # | Section/Line | Finding | Severity | Recommendation | Verification |
|---|---|---|---|---|---|
| 1 | §8.5, line 333 | **Roles cannot be defined in `wp-config.php`.** The text states: "Define user roles and capabilities in code (`wp-config.php` or a must-use plugin)." WordPress's roles and capabilities API (`WP_Roles`, `add_role()`, `add_cap()`) is initialized on the `init` hook, which runs after `wp-config.php` has been loaded and executed. Calling the roles API in `wp-config.php` will fail silently or throw a fatal error because WordPress core has not yet bootstrapped. This is also architecturally incorrect: `wp-config.php` is a configuration file for database credentials and constants, not for WordPress API calls. | Critical | Change to: "Define user roles and capabilities in a must-use plugin rather than in the database, making them resistant to SQL injection attacks and privilege escalation." Remove `wp-config.php` from this claim entirely. | Verified against WordPress core bootstrap sequence: `wp-config.php` loads at `wp-load.php` stage; `WP_Roles` is instantiated in `wp-settings.php` after plugins are loaded. Source: WordPress core source, `wp-settings.php`. |

## Notes for Cross-Document Work

**Version alignment:** The Hardening Guide declares "WordPress as of version 6.9 (2026)" (§1, line 15). The Benchmark covers "WordPress 6.x" without specifying 6.9. The Runbook template uses "6.4+" as a placeholder. No hard conflict, but @AuditAgent should verify that the 6.9 version claim is consistent with any version-specific claims in the Benchmark and Runbook, particularly around bcrypt (introduced in 6.8) and any 6.9-specific features referenced in the Hardening Guide.

**Argon2id PHP version floor:** The document correctly states "PHP 7.3+" as the floor for Argon2id via the `sodium` or `argon2` extension (§A04, line 76). This is accurate and consistent with the Benchmark. No finding raised — cited here to confirm this was checked.

**Control classification alignment (Section 15.6 matrix vs. Benchmark):** The Hardening Guide's §15.6 cross-document matrix covers File editor/mods, REST API, XML-RPC, and File ownership. The Benchmark classifies TLS 1.2+ enforcement as Level 2 (§1.1). The Hardening Guide presents this as a general recommendation without Level 1/Level 2 qualification, which is appropriate for an architecture document — the Hardening Guide does not use L1/L2 terminology. No conflict found.

**No deprecated constants without annotation:** The Hardening Guide's §15.5 guardrail list correctly annotates `FORCE_SSL_LOGIN` (deprecated), `DISALLOW_PLUGIN_EDITING`, `DISALLOW_PLUGIN_ACTIVATION`, and `SECURE_LOGGED_IN_COOKIE` (all non-core constants), and `XMLRPC_REQUEST` (read-only internal constant). The body of the document correctly uses only valid constants. No finding raised.

**Compliance language:** The document does not claim WordPress software is compliant with any framework. All compliance references are correctly attributed to deployments or organizations. Section 12.5 correctly says "WordPress deployments ... must comply with" GDPR/CCPA. Section 13 correctly attributes SOC 2 / PCI DSS certifications to hosting providers, not to WordPress. No violation found.

**Code examples:** The Hardening Guide has no fenced code blocks, consistent with the AGENTS.md architecture-level constraint. Inline code spans (backtick-formatted constants and function names) are used correctly throughout. No markdown escaping errors found in inline code.
